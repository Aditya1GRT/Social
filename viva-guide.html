Project Overview
Q: What is The Social Scoop?

A full-stack social media web application supporting user profiles, image/video posts, real-time messaging, peer-to-peer voice and video calls, follow requests (with approval), notifications, and dark/light themes — deployed as a single service on Render.com.

Database
Q: What database do you use and why MongoDB?

Primary: MongoDB Atlas (cloud, M0 free tier)

  • Document-oriented — fits social media's varied content (posts with/without media)
  • Arrays as first-class citizens — perfect for followers[], likes[], comments[]
  • $push, $pull, $in operators map naturally to social graph operations
  • Horizontal scaling path from M0 free → M2 → M10 without code changes
  • Atlas handles backups, replication, and connection pooling

Local Dev Fallback: NeDB (@seald-io/nedb 4.0.4)

  • Embedded, zero setup — data stored as .db files in /backend/data/
  • Same MongoDB-compatible query API
  • Developers run the full stack with just npm install && node server.js
Q: Explain the adapter pattern in db.js.

backend/db.js exports a MongooseAdapter class that wraps Mongoose models behind the same async API as NeDB. Every route file imports { users, posts, notifications } and calls .findAsync(), .findOneAsync(), .updateAsync() — identical calls work on both databases. The adapter handles:

  • _norm(doc): Converts Mongoose ObjectId → plain string
  • _q(query): Normalizes query syntax differences between MongoDB and NeDB
  • Connection buffering: 8s timeout, 5s auto-retry loop
  • Supports: $or, $in, $push, $pull, $set

Storage
Q: How do you handle media uploads?

Three-layer design:

  • Layer 1 — multer (memory storage): File arrives as req.file.buffer — never touches disk. 25 MB limit enforced at the middleware level.
  • Layer 2 — Storage service (backend/services/storage.js): Checks CLOUDINARY_URL env var:
    • If set → streams buffer to Cloudinary via uploader.upload_stream({ resource_type: 'auto', folder: 'social-scoop' }) → returns result.secure_url
    • If not set → writes to /backend/uploads/ with timestamp prefix → returns http://host/uploads/filename
  • Layer 3 — URL in DB: Returned URL stored in post.postMedia, message.messageMedia, or user.profilePicture
Q: Why Cloudinary?
  • CDN delivery — files served from edge nodes globally
  • Automatic format optimization (WebP for images, reduces bandwidth)
  • resource_type: 'auto' handles both images and video in one upload call
  • Generous free tier: 25 GB storage + 25 GB bandwidth/month
  • Clean delete API via uploader.destroy(publicId, { resource_type })
Q: How does file deletion work?

The deleteFile(url) function in storage.js:

  1. Extracts public_id from Cloudinary URL via regex: /\/upload\/(?:v\d+\/)?(.+)\.[^.]+$/
  2. Extracts resource_type (image/video/raw) from URL
  3. Calls cloudinary.uploader.destroy(publicId, { resource_type })

Called in: post deletion, user account cascade delete (all post media + profile picture). Both are fire-and-forget — media cleanup does not block the HTTP response.

Authentication & Security
Q: How does authentication work end-to-end?

Registration / Login:

  1. Password → bcrypt.genSalt(10) → bcrypt.hash(password, salt) → stored in DB
  2. On login: bcrypt.compare(input, stored_hash) — constant-time, safe against timing attacks
  3. JWT signed: jwt.sign({ id, username }, JWT_SECRET, { expiresIn: '30d' }) → returned to client

Token Storage & Transport:

  • Stored in Redux state → persisted to localStorage['persist:root'] by redux-persist
  • Axios request interceptor reads it and adds: token: Bearer <jwt> to every request

Server-side verification middleware:

const header = req.headers.token || req.headers.authorization;
// Accepts "Bearer <jwt>", "Barer <jwt>" (typo tolerance), or bare token
jwt.verify(raw, JWT_SECRET, (err, decoded) => {
  if (err) return res.status(403).json({ message: 'Token invalid' });
  req.user = decoded; // { id, username }
});
Q: Why JWT over sessions?

JWT is stateless — no session store needed on the server. Ideal for single-server deployment where multiple instances sharing session state is not yet set up. Trade-off: tokens cannot be revoked before expiry without a blocklist — mitigated by client-side clear on 401/403 response.

Q: Why bcrypt over SHA-256 or MD5?

bcrypt is adaptive and slow by design. At 10 rounds: ~100ms per hash — imperceptible to users, but makes GPU brute-force attacks (billions of hashes/second with SHA-256) extremely expensive. MD5 and SHA-256 are cryptographic hashes, not password hashing functions — they have no built-in work factor or salting.

Q: How do you handle token expiry gracefully?

Axios response interceptor:

api.interceptors.response.use(
  (res) => res,
  (err) => {
    if (err?.response?.status === 401 || err?.response?.status === 403) {
      localStorage.removeItem('persist:root');
      window.location.href = '/login';
    }
    return Promise.reject(err);
  }
);

Result: expired tokens trigger an automatic logout and redirect — no silent failure or broken UI.

Real-Time Communication
Q: How does real-time messaging work?
  1. On app load, client emits addUser with userId → server stores { userId, socketId } in an in-memory array
  2. Server broadcasts updated activeUsers list to all clients (powers online presence indicators)
  3. When user sends a message: saved to DB → client emits sendMessage → server relays as getMessage to recipient's socket ID
  4. Recipient's UI updates instantly via socket event listener — no polling
Q: How do voice/video calls work? Explain WebRTC.

WebRTC is a browser API for peer-to-peer media streams. Socket.IO is used only for signaling (exchanging connection metadata). Actual audio/video travels directly between browsers.

CALLER SERVER (Signal Relay) CALLEE | | | |-- getUserMedia() ------> | | |-- createPeerConnection() | | |-- createOffer() -------> | | |-- setLocalDescription -> | | |-- emit 'callUser'{offer}> |-- emit 'incomingCall' -> | | | |-- getUserMedia() | | |-- createPeerConnection() | | |-- setRemoteDescription(offer) | | |-- createAnswer() | | |-- setLocalDescription(answer) | |<- emit 'answerCall' ---| |<- emit 'callAccepted' --| | | |-- setRemoteDescription(answer) | | |<----- ICE candidates exchanged -----|------> | |========= P2P AUDIO/VIDEO STREAM DIRECT TO PEER =========|

ICE Servers used: Google STUN servers (stun:stun.l.google.com:19302, stun1.l.google.com:19302). STUN helps peers discover their public IP/port through NAT. TURN (relay) is not configured — a known limitation.

State Management
Q: Why Redux Toolkit and not Context API?

Redux Toolkit gives predictable, centralized state with DevTools time-travel debugging. createSlice eliminates action/reducer boilerplate. Context API re-renders the entire subtree on any change — fine for theme, problematic for high-frequency updates like notifications or post feeds. redux-persist syncs to localStorage so state survives page refresh without re-fetching.

Q: What is persisted vs not persisted, and why?
SlicePersisted?Reason
userYESKeep login session across refreshes — contains accessToken
conversationYESCache conversation list to avoid reload on navigation
postNOAlways fetch fresh feed — stale posts misleading
profileNOAlways fetch fresh profile — follower counts must be live
notificationsNOAlways fetch latest — unread status must be accurate
Q: Name the Redux slices and their state shape.
  • userSlice: { currentUser: null, isFetching: false, error: false }
  • postSlice: { posts: [], isFetching: false, error: false }
  • conversationSlice: { conversations: [], isFetching: false, error: false }
  • profileSlice: { user: null, posts: [], isFetching: false, error: false }
  • notificationSlice: { items: [], isFetching: false, error: false }

Deployment
Q: How is the app deployed?

Single Render.com web service using a Blueprint spec in render.yaml:

buildCommand: cd backend && npm install && cd ../app && npm install && npm run build
startCommand: node backend/server.js
healthCheckPath: /api/health

Express serves three things from one URL:

  • /api/* → Express route handlers
  • /uploads/* → Static local file fallback
  • /* catch-all → /app/dist/index.html (React SPA)

No CORS needed in production — API and frontend are the same origin.

Q: How does client-side routing work alongside Express?

API routes are registered before the catch-all in Express. So /api/posts/... hits the Express handler; any other path (like /user/john or /messages) falls through to the catch-all which serves index.html. React Router then reads the URL in the browser and renders the correct component — no full page reload.

Unique Technical Patterns
Q: Explain the self-healing database pattern.

When a user is deleted directly from MongoDB Atlas (bypassing the app), their ID remains in other users' followers[] and following[] arrays. The GET /user/:username route now:

  1. Collects all IDs from all 5 social arrays (followers, following, followRequests, reqSent, reqRecieved)
  2. Batch-checks existence with one Promise.all
  3. Filters out dead IDs → if any removed, fires a background updateAsync to fix the DB
  4. Returns cleaned data in the same response

Result: after one profile visit, the DB is permanently corrected — no cron job or migration script needed.

Q: What is stripOrphans() and why is it needed?

When a user is deleted, their like/comment entries can remain on other users' posts (orphaned references). stripOrphans() in posts.js:

  1. Collects all unique userId values from likes and comments across an entire post list
  2. Does one existence check per unique ID (not N×M per post per interaction)
  3. Filters out dead references and auto-saves if inconsistencies found

Called on both GET /profile/:userId and GET /:userId (feed). Self-heals the DB on first load after a deletion.

Q: Explain cascade delete — all 6 steps.

When DELETE /api/users/:id is called:

  1. Delete posts + media: Find all user's posts → delete each post's Cloudinary file → remove post document
  2. Clean interaction history: Find all other posts → filter this user's likes and comments → update in-place
  3. Clean social graph: Find all other users → filter this ID from followers/following/followRequests → update
  4. Delete notifications: Remove all notifications where userId or fromUserId matches
  5. Delete profile picture: Call deleteFile(user.profilePicture) to remove Cloudinary asset
  6. Delete user record: users.removeAsync({ _id: id })

Result: a deleted account leaves zero trace across all 5 collections.

Q: How do videos auto-pause when scrolled out of view?

Uses the native browser IntersectionObserver API — fires a callback when element visibility changes, with no scroll event listeners:

const observer = new IntersectionObserver(
  ([entry]) => { if (!entry.isIntersecting) el.pause(); },
  { threshold: 0.25 }  // fire when <25% visible
);
observer.observe(videoRef.current);

The 0.25 threshold means "pause when less than 25% of the video is in the viewport." No performance overhead compared to scroll listeners.

Q: What is optimistic UI and where is it used?

The UI updates immediately before the API call completes, then reverts on failure. Used for likes and comments:

// Immediate update
setLikes(nowLiked ? [...likes, userId] : likes.filter(id => id !== userId));
// Background API call — revert if it fails
reactPost(dispatch, post, userId).catch(() => {
  setLikes(liked ? [...likes, userId] : likes.filter(id => id !== userId));
});

Result: zero-latency interactions with automatic error recovery. No loading spinner on simple actions.

What Is New / Innovative
Dual-DB Adapter

Single codebase runs MongoDB in production and NeDB locally — no code changes. Most tutorials pick one. This enables zero-config local dev and production-grade DB simultaneously.

Self-Healing DB on Read

Data integrity maintained lazily on every request — not via scheduled jobs or migrations. Scales to zero maintenance overhead.

Native WebRTC Calls

Voice and video calls built into the social feed — no Twilio, Agora, or Zoom SDK. Pure browser WebRTC APIs with Socket.IO signaling.

Cascading Media Delete

Most apps leave orphaned media in storage. This app deletes Cloudinary assets on account/post deletion — clean storage, no cost leakage.

JWT Expiry UX

Instead of silently breaking when tokens expire, a 401/403 response clears session and redirects to login. Users always know why they were signed out.

IntersectionObserver Videos

Videos auto-pause when scrolled out of viewport using a native browser API — no scroll listeners, no performance hit on feeds with many videos.

Known Limitations (Honest Trade-offs)
Examiners respect self-awareness. Knowing your limitations demonstrates engineering maturity.
LimitationRoot Cause / Trade-off
No TURN serverWebRTC calls may fail on strict corporate NAT/firewalls — STUN works for ~85% of networks but not all
JWT cannot be revokedStateless design — no token blocklist; a stolen token is valid until expiry (30d)
activeUsers in memoryOnline presence list lost on server restart; not shared across multiple instances — needs Redis for scale
bcrypt on main threadBlocks Node.js event loop for ~100ms per hash on login — acceptable at small scale, needs worker threads at high load
Comment ID is Date.now()Timestamp-based IDs can collide under high concurrency — should use UUID or DB-generated IDs
No rate limitingMissing express-rate-limit — API endpoints can be abused
NeDB data lost on restartLocal dev only — acceptable trade-off, not a production concern
No E2E encryptionMessages are readable by server admin (MongoDB Atlas) — needs Signal Protocol for true privacy
